# Opus の振る舞いを Fable に近づける

**本プロジェクトの目的は、[Claude Code](https://claude.com/claude-code) のメインセッションとして動く Claude Opus を、上位モデルの Claude Fable のように振る舞わせることです** — 自分でコードを書き進めて逐一許可を求めるのではなく、設計し、実装をサブエージェントに委譲し、その結果を監査する、**アーキテクト / レビュア / サブエージェント管理者**として。(Fable は能力面で Opus の一つ上のティアに位置する Claude モデルです。)

Fable はそのリードエンジニア的な規律を自力で発揮します。Opus は素の能力はあるものの、同じ振る舞いを指示なしに確実に**発火**させません。この [`CLAUDE.md`](./CLAUDE.md) は、その差を埋める足場です。

一本の明確な弧として作られており、本READMEもそれを辿ります:

**目的** → **方法**(Fable–Opus の差を測る)→ **分析でわかったこと**(差の正体)→ **イテレーション**(差を埋めるプロンプトの開発)→ **スコア**(4つの測定系で検証)。

## 1. 目的

Opus をメインセッションとして、Fable のように振る舞わせる: **設計し、委譲し、監査する — 自分で実装して聞く、ではなく。** 下記で特定した具体的な振る舞いの差を埋めつつ、単発タスクで Opus がすでにできていることは全て保つ。

## 2. 方法 — どうやって差を測ったか

推測ではなく「*何を*変えるべきか」を決めるため、Fable–Opus の差を3つの独立した証拠源から三角測量し、いずれも同じ結論に収束させました:

1. **自然実験。** 同じ開発者が、同系のプロジェクトを一度は Fable のメインセッションで、一度は Opus のメインセッションで走らせました。Opus の回は数十件のユーザー訂正のカタログを生みました — 許可を求める繰り返し、近似的な実装、検証のユーザーへの押し戻し、リライトでの回帰。Fable の回にはこれらが皆無でした。これが「*何を直すべきか*」の直接証拠です。
2. **公開されたティア差。** 同一プロダクトのモデルティア間で、リーク/公開されたシステムプロンプトを比較すると、上位ティア(Fable 側)には下位ティア(Opus 側)にない自律性・ターン終了の規律が明示的に与えられていました — ベンダー自身が、自然実験で「欠けている」と判明した振る舞いを足場として組んでいたのです。その記述をここに輸入しました。
3. **Fable のセッションの採掘**により、移植する価値のある具体的振る舞いを抽出しました: 良い委譲ブリーフの型、サブエージェント出力に対する監査チェーン、サブエージェントの逸脱を「指示との一致」ではなく「*原則との一致*」で裁く習慣。

## 3. 分析でわかったこと — 差の正体

2つの発見が設計全体を駆動しました。

**発見1 — 差は能力ではなく、規律の発火にある。** 単発の新鮮なタスクなら、Opus は Fable に肉薄します: コードを監査し、完了を検証し、アーキテクチャを設計し、大規模コードベースを高い水準で理解します。違うのは*どの規律が指示なしに発火するか*です。Fable と比べ、メインセッションとしての Opus は次の傾向を持ちます:

- **バッチの境目ごとに止まって許可を求める**(「続けてよいですか?」)。区切れたタスクを完遂まで走り切らない。
- 型チェックが通っただけで**完了を宣言**し、ランタイム挙動の確認を*あなた*に丸投げする。
- **最初に見つけた原因でデバッグを打ち切る**。残った症状を合理化し、全レイヤーを確認しない。
- 些末な内部選択では許可を求める一方、**利用者に見える破壊的変更(文書化されたAPIのリネーム)を無断で行う** — エスカレーションの軸が逆転している。
- 長いセッションで**規律が減衰する**: 序盤に述べたルールが、コンテキストが埋まるにつれて崩れる。

**発見2 — したがってプロンプトは、志ではなく手続きでなければならない。** 能力はあるが反射がないため、原則リスト(「丁寧にやれ」)は、**発火条件つきの手続き**(「Xを受け取ったら、Yする前に、Zせよ」)より測定上明確に劣ります。そして最も崩れやすい規律は3通り — 原則・名指しのアンチパターン・自己点検 — で述べています。最高頻度の失敗モードへの冗長性こそ、ベンチマークを動かしたものだからです。

## 4. イテレーション — プロンプトの開発

プロンプトは一発書きではなく、測定ループとして開発しました:

1. **ベースライン。** 標準の Opus(および汎用 `CLAUDE.md` 付きの Opus)を、判別タスクのバッテリー — 監査 / 自律完走 / 意図読解 / 完了検証 / 委譲仕様 / 多層デバッグ — で採点する。
2. **仮説。** 観測された各ギャップに1つずつ仮説を立て、それぞれ §2 の証拠に紐付ける。
3. **執筆と再採点。** 仮説を発火条件つきの手続きとして符号化し、再採点する。判別タスクを**残りに退行を出さずに**動かした変更だけを残す。
4. **過学習の防止。** プロンプトを一度もチューニングしていないホールドアウトタスク(白紙からのアーキテクチャ設計、挙動保存リファクタ)で再確認する。
5. **収束。** バッテリーが飽和し、ホールドアウトが退行を示さなくなったら止める。

規律は、標準の Opus が実際に Fable と乖離した3タスク(多層デバッグ、エスカレーション較正、意図読解)を狙いました。監査 / 完了検証 / 設計のタスクは標準の Opus ですでに天井にあり、そこを保ちました。

## 5. スコア — 4つの測定系での検証

汎用の標準 `CLAUDE.md`(およびプロンプトなしのベースライン)と比較テストし、可能な限り客観的な採点器を用いました — 隠しテストスイート、結果リポジトリ状態の独立再実行、バイト単位の出力比較、引用の抜き取り検証。

| 測定系 | 測るもの | 結果 |
|---|---|---|
| **自作ベンチマーク・バッテリー** | 監査 / 自律完走 / 意図読解 / 完了検証 / 委譲仕様 / 多層デバッグ | 本プロンプトで **90 / 90**、標準プロンプトで **77 / 90**。利得は3つの判別タスクに集中し、残りに退行なし |
| **実OSSタスク**(バリデーションライブラリ / HTTPフレームワーク / ビルドツール) | 実際にマージ済みのissue修正、大規模コードベースの理解 | 実マージPRの回帰テストを隠し採点器として使用。標準プロンプト比で**退行ゼロ**。大規模コードベースのアーキテクチャ地図は引用を一つずつ検証し、捏造ゼロ |
| **実Claude Codeセッションでのフラットベンチ** | 同じタスクを、模擬でなく本物のメインセッションで実行 | Opus + 本プロンプトは、判別タスクで **Fable と同等**に到達し、標準の Opus を明確に上回った |
| **長丁場マラソン**(8ウェーブ連続、約100万トークン処理) | 規律が疲労と肥大するコンテキストで生き残るか | 多層デバッグとエスカレーション較正の優位は**疲労下でも保存**された: 後半ウェーブで、標準プロンプトは2つ目のバグを合理化して見逃したが、本プロンプトは両層を発見して修正した |

**正直な但し書き**(本ドキュメント自身の報告規律に倣って): いくつかのセルは単発実行であり、採点者が実験設計者でもあった(全スコアを客観的・独立再実行可能なチェックに落とすことで緩和)。そして最も極端な実世界の失敗(数日のセッションでの数十回の許可要求)は、固定長テストが届かない「数日・複数回のコンテキスト要約」規模を要します。長丁場対策は証拠に基づきますが、その極端な規模での挙動は実運用に委ねています。

## プロンプトが実際に足すもの

主要な介入と、それぞれが対応する観測された Fable–Opus の差:

- **自律完走(§0.1, §5)。** 依頼から導かれる可逆な作業は聞かずに進める。報告を「続行の許可待ち」で締めない。
- **行動の前に意図(§0.2)。** 誤字・言い差し・別コンテキスト宛てに見える指示は、字面でなく意図で読む。名前と挙動が食い違うときは、名前だけでなく*意味の問題を全層で*直す。
- **三車線のエスカレーション分類器(§2)。** すべての判断を、*まず聞く*(利用者に見える契約を壊す)/*仮定を述べて進む*(原則から導ける実行詳細)/*主張の前に検証する*(完了主張)に分類 — 逆転したエスカレーション軸を直接直すため。
- **デバッグ降下プロトコル(§4.3)。** 何かに触れる前に「正しい期待状態」を定量化する。コードだけでなく全レイヤーを疑う。1つ原因を見つけても探索を終えない。
- **「完了」の語彙ゲート(§4.5)。** *done / passing / 完了 / 完成* の語は、自分で検証軸を実行した後にのみ書ける。結果の観測をユーザーに求めるのは、一段スキップしたのと同じ。
- **委譲ブリーフのテンプレートと監査チェーン(§4.9)。** どの委譲ブリーフにも必須の8セクション、結果に対して回す固定の監査手順 — サブエージェントの「全green」自己申告は監査の*入力*であって、その結論ではないから。
- **長丁場の減衰対策(§4.10)。** ウェーブ境界での再アンカー、恒久ルールをメモリでなくファイルに書くこと、長い機械的作業のための精度減衰の自己点検。

これ以外の部分は、上記の介入が乗る土台となる一般的なエンジニアリング哲学 — 8つのメタ原則、ベンダー分離、ハードカットオーバー規律、fail-closed 防御、ブループリント対履歴のドキュメント規則 — を担っています。

## 使い方

Claude Code で駆動するリポジトリのルートに [`CLAUDE.md`](./CLAUDE.md) を置き、メインセッションのモデルを Opus にします。プロジェクトの構成・コマンド・ドメイン知識は、末尾の **Project-Specific** 節に追記してください。

いくつかの注意:

- **サブエージェント対応のハーネスを前提とします。** モデル戦略(§1)は、通常の実装をサブエージェントに委譲し、メインセッションを設計・監査・レビューに留めます。サブエージェントがない環境でも、設計・監査の規律はそのまま適用でき、委譲の各節は「*あなた自身*がその作業をどう構造化し検証するか」の指針になります。
- **モデル名は可能な限り抽象化されています。** 本文中でティアが名指しされている箇所は差し替えが容易です。意図は特定のプロダクト名ではなく、推論ティアの抽象(light / middle / heavy)です。
- **例示は3言語です。** 例として引かれる語句は英語 / 日本語 / 中文を交換可能に用いています。規律そのものは言語非依存です。あなたのプロジェクトの作業言語に置き換え・拡張してください。
- **これはブループリントであり、変更履歴ではありません。** それ自身の §6 に従い、常に現在形に保ちます。ルールが変わったら、履歴を追記するのではなく、その節を新しい現在の状態として書き直してください。

## 設計ノート

- 本ドキュメントは意図的に「発火条件つきの手続き」として書かれています。中心的発見は「Opus はこれらを*できる*が、指示なしには確実に*発火しない*」だからです。原則リストは「Xを受け取ったらYせよ」より測定上明確に弱いものでした。
- 元資料で最も訂正された振る舞い — バッチを許可要求で締めること — は、意図的に3箇所(§0.1, §2, §5)で扱っています。最高頻度の失敗モードへの冗長性は、ベンチマークを動かした技法の一つでした。
- エスカレーション分類器が単純な「迷ったら聞け」ではなく三車線なのは、観測された失敗が*聞かなさすぎ*ではなく*間違ったことを聞く*ことだったからです。逆転の両方向を名指しする必要がありました。
